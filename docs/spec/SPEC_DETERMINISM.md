# 输出稳定性与规范化规范（v1）

本文档定义“同一组输入 -> 同一份输出”的稳定性要求，以及节点去重、派生节点生成、命名、排序规则。

目标：让生成结果可 diff、可回滚、可定位问题；避免因为上游顺序抖动导致输出大面积噪音。

---

## 1. 名词与约束

### 1.1 命名空间

在 Surge/Shadowrocket 等目标中，节点名与策略组名通常共享同一命名空间（规则/组引用都用字符串）。

因此 v1 约束：
- 用户定义的策略组名必须全局唯一。
- 自动诊断组名 `CHAIN-<custom_proxy.name>` 也属于同一命名空间。
- 最终策略组名不得与任何最终节点名冲突；若冲突必须报错。

### 1.2 保留名与保留前缀

以下字符串作为内置 action 保留（大小写敏感，v1 以大写为准）：
- `DIRECT`
- `REJECT`

以下前缀保留给自动诊断组：
- `CHAIN-`

约束：
- 用户定义的策略组名不得为保留名。
- 用户定义的策略组名不得以 `CHAIN-` 开头。
- 节点名若为保留名，允许编译器进行确定性重命名（见 3.3）。

### 1.3 节点身份与展示名

v1 将节点的“内部身份”与“外部展示”分离：
- 每个编译后的 Proxy 都必须拥有一个内部唯一标识 `proxyID`。
- `proxyID` 仅用于编译阶段与渲染阶段的内部关联，不进入 profile 语法、不进入模板锚点语义，也不直接出现在最终配置文件与 `mode=list` 输出中。
- `Name` 继续作为节点的展示名与正则匹配对象；其唯一化目标是保证最终配置可表示，而不是承担内部主键职责。

---

## 2. 节点分类

最终编译产物中的节点分两类：

1. 原始订阅节点
- 来自订阅解析与去重结果。
- v1 订阅输入仍只产生 `type=ss`。

2. 链式派生节点
- 来自 `custom_proxy + proxy_chain`。
- 表示“通过某个原始订阅节点访问某个自定义代理模板”的可输出节点。
- 可为 `ss/http/https/socks5/socks5-tls`。

约束：
- `custom_proxy` 本身不直接进入最终输出。
- 派生节点一旦生成，就与原始订阅节点一起进入最终 `Proxies[]`，供后续策略组正则筛选与渲染使用。

---

## 3. 原始订阅节点规范化

### 3.1 订阅合并顺序（稳定输入序）

当请求包含多个订阅 URL：
- 合并顺序：按请求参数顺序合并（GET 的 `sub=` 出现顺序；POST 的 `subs[]` 数组顺序）。
- 每个订阅内部：按文本行顺序解析得到 Proxy 列表。

该“合并顺序”是后续去重与命名冲突处理的稳定基准。

### 3.2 字段规范化

对每个原始订阅节点，v1 至少应做以下规范化（不改变语义）：
- `Name`：去首尾空白；不得包含 `\r`、`\n`、`\0`。
- `Server`：去首尾空白；域名应转小写（IP 原样）。
- `Cipher`：去首尾空白并转小写。
- `Password`：去首尾空白。
- `Plugin`/`PluginOpts`：去首尾空白（不做重排）。

### 3.3 `proxyID` 生成

对每个规范化后的原始订阅节点，v1 必须生成稳定的 `proxyID`：
- 输入：使用《4.1 去重 key（SS）》定义的规范化语义 key。
- 算法：对该 key 做稳定哈希；v1 推荐使用完整 SHA-256 十六进制字符串。
- 时机：`proxyID` 生成发生在去重与最终命名之前。

约束：
- 同一语义节点在不同请求、不同进程重启后，只要规范化后的语义 key 一致，`proxyID` 就必须保持一致。
- `proxyID` 不依赖展示名生成规则；展示名后缀变化不应影响 `proxyID`。

### 3.4 节点命名生成与冲突处理

对每个原始订阅节点先得到一个“基础名” `baseName`：
- 若订阅提供 `#name`：URL 解码后得到的名称，去首尾空白，作为 `baseName`。
- 否则：使用 `"<server>:<port>"` 作为 `baseName`。

基础名必须做最小兼容性规范化：
- 将所有 `=` 字符替换为 `-`（避免 Surge 系列配置解析失败；该处理必须是确定性的）。

然后按“合并顺序”依次为每个节点分配最终 `Name`，保证全局唯一：
- 若 `baseName` 未被占用且不是保留名：使用 `baseName`。
- 若 `baseName` 已被占用，或 `baseName` 是保留名：追加后缀 `-2`、`-3`…，直到找到未占用的名字。

## 4. 原始订阅节点去重（Deduplication）

### 4.1 去重 key（SS）

对 SS 节点定义去重 key，建议包含：
- `Type`
- `Server`
- `Port`
- `Cipher`
- `Password`
- `Plugin`
- `PluginOpts`

该语义 key 同时服务于：
- 去重判断
- 原始订阅节点 `proxyID` 生成

### 4.2 去重策略

- 保留第一次出现的节点（按“合并顺序”）。
- 后续重复 key 的节点直接丢弃。
- 被保留的节点保留其对应的 `proxyID`；后续重复节点不再进入命名分配与组成员展开。

注意：
- 去重发生在“最终命名分配”之前。

---

## 5. 派生节点生成

### 5.1 派生节点的输入关系

派生节点由二元关系唯一确定：

```
(custom_proxy, subscription_proxy)
```

其中：
- `custom_proxy`：profile 中定义的基础代理模板
- `subscription_proxy`：被 `proxy_chain` 选中的原始订阅节点

同一个 `custom_proxy` 可由多条 `proxy_chain` 规则命中；最终命中集合按原始订阅节点 `proxyID` 做并集去重。

### 5.2 派生节点 `proxyID` 生成

对每个派生节点，v1 必须生成稳定的 `proxyID`：
- 输入：`custom_proxy` 的规范化语义 key + `ViaProxyID`
- 算法：稳定哈希；v1 推荐使用完整 SHA-256 十六进制字符串

约束：
- 派生节点的 `proxyID` 不依赖最终展示名。
- 同一 `custom_proxy` 通过同一原始订阅节点生成的派生节点，在不同请求中 `proxyID` 必须保持一致。

### 5.3 派生节点命名

派生节点的基础名固定为：

```
<custom_proxy.name> via <subscription_proxy.Name>
```

命名规则：
- 按 `custom_proxy` 声明顺序处理。
- 同一 `custom_proxy` 下，再按命中的原始订阅节点输出顺序处理。
- 若基础名与已占用名称冲突，则追加后缀 `-2`、`-3`…，直到找到未占用的名字。

说明：
- 名称占用集合同时包括原始订阅节点、先前生成的派生节点与最终策略组名。
- 若自动诊断组名与最终节点名冲突，必须报错，而不是静默改写组名。

### 5.4 派生节点顺序

最终 `Proxies[]` 的顺序固定为：
1. 全部原始订阅节点（按订阅合并顺序）
2. 全部派生节点：
   - 先按 `custom_proxy` 声明顺序
   - 再按命中的原始订阅节点顺序

最终输出不再额外排序。

---

## 6. 策略组编译顺序与递归展开

### 6.1 用户定义策略组

- 用户定义的策略组顺序必须与 profile 的 `custom_proxy_group` 列表顺序一致（不排序）。
- 编译后的策略组成员必须是类型化引用：
  - `proxy`：引用某个 `proxyID`
  - `group`：引用某个策略组名
  - `builtin`：引用 `DIRECT` 或 `REJECT`

### 6.2 `@all` 与正则组

- `@all` 展开为全部原始订阅节点引用，不包含派生节点。
- `select/url-test` 的正则写法从“最终可输出节点”中筛选成员，因此包含派生节点。

### 6.3 `proxy_chain type=group` 的递归展开

当 `proxy_chain.type=group` 时，递归展开规则为：
- 遇到 `proxy` 成员：收集对应的原始订阅节点 `proxyID`
- 遇到 `group` 成员：继续递归展开
- 遇到 `builtin` 成员：忽略

约束：
- 递归展开只针对“用户定义策略组”。
- 自动诊断组不属于 `proxy_chain type=group` 的展开对象。
- 递归过程中若发现组引用环，必须报错（`GROUP_REFERENCE_CYCLE`）。

### 6.4 自动诊断组顺序

自动诊断组顺序固定为：
- 先输出全部用户定义策略组
- 再按 `custom_proxy` 声明顺序输出自动诊断组 `CHAIN-<custom_proxy.name>`

自动诊断组成员顺序与对应派生节点顺序一致。

---

## 7. 规则输出顺序

最终规则列表顺序必须严格为：
1. 按 profile 的 `ruleset` 列表顺序逐个插入
2. 再按 profile 的 `rule` 列表顺序追加 inline 规则

规则不做任何排序或去重，以保证“用户写的顺序就是最终顺序”。

---

## 8. `mode=list` 输出稳定性

当 `mode=list` 输出纯节点列表时：
- 只输出原始订阅节点；不输出派生节点。
- 节点列表顺序：按原始订阅节点输出顺序（不额外排序）。
- raw（明文）输出：每行输出一条 canonical 的 `ss://` URI；使用 `\n` 分行，并且末尾必须带一个 `\n`。
- 当 `encode=base64`：对 raw 列表文本做标准 base64 编码输出；不得换行折行。

说明：
- `mode=list` 的语义保持“只列出订阅节点”，不把 profile 派生能力混入该模式。

---

## 9. Surge `#!MANAGED-CONFIG` 的 URL 稳定性

当 `mode=config&target=surge` 时需要生成 `<CURRENT_CONVERT_URL>`。

为保证稳定，v1 要求：
- 无论请求是 GET 还是 POST，`<CURRENT_CONVERT_URL>` 都必须由服务端根据“解析后的参数”重新序列化生成，不得直接复用原始 query 字符串。

- base URL 选择：
  1. 若 profile 提供 `public_base_url`，必须使用它作为 base URL。
  2. 否则使用当前请求推导出的 base URL。

- query 参数序列化顺序（固定）：
  1. `mode=config`
  2. `target=surge`
  3. `fileName=<name>`（可选）
  4. 按请求中订阅数组顺序重复输出 `sub=<url>`
  5. `profile=<url>`
